# Contributing

Contributions are welcome, but this repo is intentionally selective.

Before opening a pull request, read [CONTENT_STANDARD.md](./CONTENT_STANDARD.md).

## Good First Contributions

- Add a missing failure mode to an existing pattern
- Add a source to support or correct a claim
- Improve an architecture diagram
- Propose a case study outline
- Add an evaluation method for an existing pattern
- Clarify a tradeoff

## Pull Request Checklist

Your PR should answer:

- What system design problem does this improve?
- What claim or recommendation is being added?
- What evidence supports it?
- What tradeoff or failure mode is included?
- What evaluation or observability method is included?

## Source Suggestions

If you are suggesting a resource, use this format:

```text
Title:
URL:
Topic:
Source type: official | research | implementation | engineering blog | practitioner
Why it matters:
Production implication:
Claim this supports:
```

## Tone

Write like an engineer explaining a design review. Be precise, concrete, and skeptical of hype.

## Review Policy

Maintainers should reject contributions that:

- Add noise without a design decision
- Repeat common AI claims without evidence
- Make vendor-specific recommendations without tradeoffs
- Describe a pattern without evaluation or failure modes
- Turn the repo into a link list
